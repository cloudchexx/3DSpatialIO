# 三维空间数据高效读写 — 开发文档

---

## 一、项目概述

设计一个**无冗余、高均衡、高性能**的三维数据存储引擎。核心目标：在不增加冗余副本的前提下，使 X、Y、Z 三个维度的切片读取速度均衡且高效。

---

## 二、核心约束

| 约束项 | 要求 |
|--------|------|
| 精度 | 单点相对误差 < 1‰（一票否决） |
| 存储膨胀 | ≤ 原始数据 1.5 倍（满分 20） |
| 内存上限 | ≤ 2GB（超过可忽略该限制，但仍需可配置） |
| 并发模型 | 单进程多线程，无多节点 |
| 语言 | C/C++ 优先，可 CUDA 加速 |
| 开放库 | 仅限开源免费库，禁商用闭源 |

---

| 评分项 | 分值 | 关键点 |
|--------|------|--------|

| 性能指标 | 60 | X/Y/Z 三维随机切片(100次) + 连续切片(10次)，读完后写出标准格式，综合用时越短越好 |

#### `include/file_io.h` + `src/file_io.cpp` — 原始文件读取器

```cpp
// 原有：小文件全量读取（保留，供测试用）
struct RawDataInfo {
    uint32_t dim_x, dim_y, dim_z;
    size_t   total_elements;
    float    value_min, value_max;
};

RawDataInfo read_raw_float_file(
    const std::string& path,
    uint32_t dim_x, uint32_t dim_y, uint32_t dim_z,
    std::vector<float>& out_data);

// 新增：流式源文件读取器（大文件支持）
class RawFileReader {
public:
    // 打开源文件并校验总字节数（用 64 位 API）
    // skip_bytes 指定文件头跳过字节数（默认 0）
    bool open(const std::string& path,
              uint64_t dim_x, uint64_t dim_y, uint64_t dim_z,
              int elem_size = 4,
              uint64_t skip_bytes = 0);
    void close();

    // 读取一个逻辑 chunk 区域，按 chunk 内部布局排列
    size_t read_region(uint64_t x0, uint64_t x1,
                       uint64_t y0, uint64_t y1,
                       uint64_t z0, uint64_t z1,
                       void* dst);

    // getter
    uint64_t dim_x() const, dim_y() const, dim_z() const;
    int      elem_size() const;
    uint64_t total_elements() const;
    uint64_t data_offset() const;
    FILE*    file_handle() const;
};
```

- 原有 `read_raw_float_file()`：一次性读入 `std::vector<float>`，供测试和小文件使用
- 新增 `RawFileReader`：流式按需读取，内存与数据总量解耦；使用 `_fseeki64` / `_ftelli64` 64 位 API，突破 2GB 文件上限；支持 `--skip-bytes` 跳过文件头
- 定义跨平台宏 `FSEEK64` / `FTELL64`（Win: `_fseeki64` / `_ftelli64`，Unix: `fseeko64` / `ftello64`）

#### `src/main.cpp` — 程序入口

流程（已适配流式架构）：
1. 解析 CLI 参数
2. `reader.open()` 流式校验文件（64 位 API，不分配全量内存）
3. 输出文件路径、维度、元素数、数据量 (MB)
4. `find_optimal_chunk_shape()` 纯几何计算（不读数据）
5. `write_c3dr_file_stream()` 流式落盘（子组为单位，内存仅 nc_z 个 chunk）
6. 可选：`--profile` 标定 W_MEM、`--benchmark` 切面读取性能回归
7. 捕获异常并输出错误信息

### 7.3 算法流程

候选池 `{16, 32, 64, 128, 256, 512}` 三层穷举 (216 种组合)，依次通过：

1. **体积约束过滤** — 单块 512KB ~ 4MB（以 float 为例），避免索引膨胀和 I/O 粒度失衡
2. **存储红线过滤（一票否决）** — Padding 膨胀率 < 1.45（留 5% 裕量），`int64_t` 防溢出
3. **内存连续性感知评分** — `cost = max(Tx,Ty,Tz) / min(Tx,Ty,Tz)`，选 cost 最小者

得到满足 $c_x > c_y > c_z$ 逆向补偿策略的最优 Chunk 形状。

## 七-B、大文件流式改造 — 已完成 ✅

> 详细方案见 [`大文件流式改造实施方案.md`](./大文件流式改造实施方案.md)

### 7B.1 背景与目标

原有架构存在 5 个 32 位/全量内存瓶颈：
1. `fseek`/`ftell` 返回 `long`（Win 32 位上限 2GB）
2. `out_data.resize(num_floats)` 一次性全量分配
3. `ftell(f)` 跟踪输出偏移（>2GB 溢出）
4. `fseek(static_cast<long>(offset))` 偏移截断
5. 全量 `vector<float> data` 绑定全流程

**改造目标**：支持 TB 级数据包，内存占用 ≤ 2GB。

### 7B.2 核心改动

| # | 模块 | 改动 |
|---|------|------|
| 1 | `include/file_io.h` | 新增 `RawFileReader` 类（流式读取器）+ 跨平台 `FSEEK64`/`FTELL64` 宏 |
| 2 | `src/file_io.cpp` | 实现 `open()`（64 位文件大小校验 + `skip_bytes` 支持）、`read_region()`（按区域流式读取）、`close()` |
| 3 | `include/chunk_io.h` | 新增 `write_c3dr_file_stream()` 声明；`C3DRIndexEntry::offset` 改为 `int64_t`；引入 `file_io.h` |
| 4 | `src/chunk_io.cpp` | 实现子组流式写入算法；修复 `ftell` → 公式计算；修复 `fseek` → `FSEEK64`；修复写入排序 bug；优化 I/O 为大块连续读 |
| 5 | `src/main.cpp` | 替换 `vector<float> data` → `RawFileReader`；流式流程：open → optimize → stream_write |
| 6 | `src/profiler.cpp` | `benchmark_slices()` 适配 `RawFileReader&`，移除全量数据依赖 |
| 7 | `src/args.cpp` | 新增 `--output`、`--elem-size`、`--skip-bytes`、`--profile`、`--benchmark` 参数 |

### 7B.3 子组流式写入算法

```
FOR iy = 0 .. nc_y-1:
    FOR ix = 0 .. nc_x-1:
        // 步骤 A：每 x 层一次大块 fread（~640KB），分散到 nc_z 个 chunk 缓冲区
        FOR lx = 0 .. x_len-1:
            fread(y_row_buf)  // 一次性读取固定 x 下 y 区间全部 z 数据
            FOR iz = 0 .. nc_z-1:
                memcpy chunk_bufs[iz] ← y_row_buf 对应片段

        // 步骤 B：FSEEK64 定位后按 chunk_idx 写入各自偏移
        FOR iz = 0 .. nc_z-1:
            fseek(data_offset + chunk_idx * chunk_bytes)
            fwrite(chunk_bufs[iz])

回写 Header + IndexTable
```

**内存占用**：仅 `nc_z × cx × cy × cz × 4` + `IndexTable`，典型 < 100 MB。

### 7B.4 过程中修复的关键 Bug

| Bug | 描述 | 修复 |
|-----|------|------|
| 写入顺序错位 | 子组遍历 `(iy, ix)` ≠ chunk_idx 线性顺序，顺序 `fwrite` 导致 chunk 数据写入错误偏移 | 步骤 B 每次写入前显式 `FSEEK64` 到正确位置 |
| I/O 性能瓶颈 | 原始逐 `(x,y)` 对随机寻址 → 18GB 文件需 ~8000 万次 seek | 改为每 x 层一次大块连续 `fread`（~640KB），降至 ~1.5 万次 seek（~5000× 提升） |

### 7B.5 18GB 实测结果
