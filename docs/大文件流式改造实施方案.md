# 大文件流式改造实施方案

> **目标**：使系统支持官方 18GB / 49GB 乃至 TB 级数据包，内存占用与数据总量解耦，全程 ≤ 2GB。

---

## 一、现状问题诊断

| # | 位置 | 问题 | 后果 |
|---|------|------|------|
| 1 | `src/file_io.cpp:20-22` | `fseek`/`ftell` 返回 `long`（Win 32 位，上限 2GB） | 18GB 文件大小校验必然失败 |
| 2 | `src/file_io.cpp:36` | `out_data.resize(num_floats)` 一次性分配全量内存 | 18GB 数据需 ~19GB 内存，多数机器 OOM |
| 3 | `src/chunk_io.cpp:84` | `ftell(f)` 跟踪输出文件偏移 | 输出 .c3dr 也可能超 2GB，溢出 |
| 4 | `src/chunk_io.cpp:186,236` | `fseek(..., static_cast<long>(offset))` | 偏移 > 2GB 时截断 |
| 5 | `src/main.cpp:57-59` | 全量 `vector<float> data` + 全量 `RawDataInfo` 统计 | 整个流程绑定在全量内存上 |

---

## 二、改造原则

1. **分块算法不动** — `find_optimal_chunk_shape()` 纯几何计算，零依赖实际数据
2. **每个源字节只读一次** — 源文件按 chunk 顺序遍历，读取后立即写入目标
3. **子组为工作单元** — 内存中只保留 `nc_z` 个 chunk 的缓冲区（~几十 MB）
4. **64 位文件 API** — 全部文件偏移操作使用 `_fseeki64` / `_ftelli64`（Win）或 `fseeko64` / `ftello64`（Unix）

## 三、数据结构设计

### 3.1 新增：RawFileReader（流式源文件读取器）

```cpp
// include/file_io.h

class RawFileReader {
public:
    // 打开源文件并校验总字节数（用 _ftelli64）
    // elem_size 默认 4（float32），支持扩展
    bool open(const std::string& path,
              uint64_t dim_x, uint64_t dim_y, uint64_t dim_z,
              int elem_size = 4);

    void close();

    // 读取一个逻辑 chunk 区域 [x0,x1) × [y0,y1) × [z0,z1)
    // 写入 dst，dst 大小必须 = (x1-x0) * (y1-y0) * (z1-z0) * elem_size
    // dst 布局：按 Z 最内层排列（与 chunk 内部布局一致）
    // 返回实际读取字节数
    size_t read_region(uint64_t x0, uint64_t x1,
                       uint64_t y0, uint64_t y1,
                       uint64_t z0, uint64_t z1,
                       void* dst);

    // ── 基本信息 ──
    uint64_t dim_x()    const { return m_dim_x; }
    uint64_t dim_y()    const { return m_dim_y; }
    uint64_t dim_z()    const { return m_dim_z; }
    int      elem_size() const { return m_elem_size; }
    uint64_t total_elements() const { return m_total_elements; }

private:
    FILE*    m_file = nullptr;
    uint64_t m_dim_x = 0, m_dim_y = 0, m_dim_z = 0;
    int      m_elem_size = 4;
    uint64_t m_total_elements = 0;
};
```

**`read_region` 实现要点**：对于源文件中 row-major 存储（X-Y-Z 顺序），`read_region` 逐 X、逐 Y 遍历，每次 `fseek` 到 `(x * dim_y * dim_z + y * dim_z + z0) * elem_size`，连续读 `(z1 - z0) * elem_size` 字节。利用文件系统缓存，实际性能接近顺序读。

## 四、子组流式写入算法

### 4.1 核心思想

传统的三重循环（ix → iy → iz）会导致对源文件的随机访问。改进方案：**以 (ix, iy) 子组为单位，缓冲一个子组的所有 chunk，一次从源文件顺序读取，一次顺序写入目标文件。**

```
算法：子组流式写入
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
输入: 源文件 reader, 输出路径 out_path, shape(cx,cy,cz)
输出: .c3dr 文件

1. 计算 nc_x, nc_y, nc_z
2. 写入 Header + IndexTable 占位
3. 分配子组缓冲数组:
      chunk_bufs[nc_z]  每个 = cx × cy × cz × 4 字节
4. FOR iy = 0 .. nc_y-1:               ← 外循环
       FOR ix = 0 .. nc_x-1:           ← 中循环
           // ── 步骤 A：从源文件读取整个子组的 nc_z 个 chunk ──
           FOR iz = 0 .. nc_z-1:
               read_region(
                   x0 = ix*cx, x1 = min((ix+1)*cx, dim_x),
                   y0 = iy*cy, y1 = min((iy+1)*cy, dim_y),
                   z0 = iz*cz, z1 = min((iz+1)*cz, dim_z),
                   dst = chunk_bufs[iz])
               chunk_bufs[iz] 剩余区域保持 0（read_region 只填有效区域）

           // ── 步骤 B：顺序写入 nc_z 个 chunk 到输出文件 ──
           FOR iz = 0 .. nc_z-1:
               计算 chunk 偏移 = data_offset + chunk_idx * chunk_bytes
               写入 chunk_bufs[iz]

5. 回写 Header + IndexTable（偏移已在步骤 4 计算好）
```

### 4.2 内存分析

| 变量 | 大小公式 | 典型值（801×2405×2501, 128×64×48） |
|------|----------|-------------------------------------|
| `chunk_bufs[nc_z]` | `nc_z × cx × cy × cz × 4` | 53 × 128×64×48×4 = ~83 MB |
| IndexTable | `nc_x × nc_y × nc_z × 8` | ~4 MB |
| 其他（Header、临时变量） | 常数 | < 1 MB |
| **合计** | | **< 100 MB** |

### 4.3 复杂度

- **时间复杂度**：每个源字节读一次、写一次，O(N)
- **磁盘 I/O**：源文件随机寻址（但在文件系统缓存覆盖下），目标文件顺序写
- **内存复杂度**：O(nc_z × chunk_volume)，与数据总量无关

<details>
<summary>open() 64 位校验示例</summary>

```cpp
#ifdef _WIN32
    #define FSEEK64 _fseeki64
    #define FTELL64 _ftelli64
#else
    #define FSEEK64 fseeko64
    #define FTELL64 ftello64
#endif

bool RawFileReader::open(const std::string& path,
                         uint64_t dim_x, uint64_t dim_y, uint64_t dim_z,
                         int elem_size) {
    m_file = fopen(path.c_str(), "rb");
    if (!m_file) return false;

    FSEEK64(m_file, 0, SEEK_END);
    int64_t file_size = FTELL64(m_file);
    FSEEK64(m_file, 0, SEEK_SET);

    m_dim_x = dim_x;
    m_dim_y = dim_y;
    m_dim_z = dim_z;
    m_elem_size = elem_size;
    m_total_elements = dim_x * dim_y * dim_z;

    uint64_t expected = m_total_elements * elem_size;
    if (static_cast<uint64_t>(file_size) != expected) {
        close();
        return false;
    }
    return true;
}
```
</details>

## 七、风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| Windows `_fseeki64`/`_ftelli64` 兼容性 | 低 | 编译失败 | `#ifdef _WIN32` 宏切换，MSVC 2015+ 均支持 |
| `read_region` 随机寻址性能差 | 中 | 导入速度慢 | 文件系统页缓存（OS 级）通常足够；可后续改为预读线程 |
| 新旧两路校验和不一致 | 低 | 数据错误 | TODO 7 对比测试覆盖 |
| `dim_x × dim_y` 乘积超 `uint64_t` | 极低 | 崩溃 | 上限 2^64 ≈ 1.8×10^19 元素，TB 级数据远不及 |
